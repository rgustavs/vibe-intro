01 · Why this matters

An agent doesn't answer. It acts.

A chatbot returns text. An agent calls APIs, queries data, files tickets, drafts emails, books meetings. Each action has a consequence your customer can see — or can't undo.

If the prompt was the spec, the toolset is now the surface area. Both are yours to own.

Chatbot
Prompt → Text

The model produces an answer. A person reads it and decides what to do.

Agent
Prompt + Tools → Actions

The model decides what to do, calls a tool, observes, and decides again — sometimes for many steps.

02 · Definition

An agent is a model in a loop.

It looks at the current state. It decides on a next step. It uses one of its tools. It observes what came back. Then it does it again — until the goal is met, or it knows to stop.

1
ObserveWhat does the current state look like?
2
DecideWhich tool should I call, with what arguments?
3
ActCall the tool. Wait for the result.
4
Repeat…until the goal is met, or stop.

A chatbot answers a turn. An agent runs a process.

03 · Anatomy

Five parts of a working agent.

Goal

What success looks like. One sentence the agent can check itself against.

Tools

The actions it can take. Each one named, scoped, and described.

Context

What it knows on start. Docs, data, prior steps, retrieved evidence.

Policy

How it chooses what to do next. The prompt — but now about behavior, not output.

Stop

When to halt. Goal met, budget spent, or stuck — each is a stop.

You don't need all five every time. But every missing piece is a failure mode waiting to happen.

Principle 1

Tools are the product.

loose Agent has access to: - our API - our database - email
scoped Agent has exactly these tools: search_orders(date_from, date_to) → top 50, read-only draft_reply(order_id, body) → draft only; never sends escalate(order_id, reason) → opens a ticket; never auto-resolves

If you can't list the tools, you can't review the agent. The toolset is the most reviewable part of the whole system — use it.

Principle 2

Decide what it can do without asking.

Questions to answer before shipping
  • Maximum steps before stopping?
  • Maximum budget before pausing?
  • Which actions require human confirmation?
  • Which actions are irreversible?
  • Who gets paged when it fails?

Autonomy is a slider, not a switch. Start low. Earn each notch with real evidence the agent behaved.

Principle 3

If you can't see what it did, you can't trust it.

Every tool call, every argument, every result — recorded. The trace is how the team debugs, how customers complain effectively, and how you find out the agent is silently broken.

step 1search_orders("2026-05-15", "2026-05-21")
→ 23 matches
step 2read_order("o_3819")
→ status: shipped, complaint about delay
step 3draft_reply("o_3819", body=…)
→ awaiting human review

Build the flight recorder before the cleverness. You will need it on day one, not month three.

Principle 4

Agents fail in new ways.

Loops

Same tool, same arguments, three times in a row. Cap retries. Detect repetition.

Wrong arguments

Hallucinated IDs, fabricated dates, made-up emails. Validate every input before the tool runs.

Runaway cost

500 API calls before anyone notices nothing's working. Step budget, token budget, hard alarm.

Silent success

Agent reports "done" — but nothing actually changed. Verify the side effect, don't trust the summary.

04 · Iteration

Eval the trajectory, not just the answer.

  • Did it pick the right tool?
  • Did it call it with sensible arguments?
  • Did it stop at the right moment?
  • Did it ask for help when it should have?
  • Two agents can reach the same answer — one by luck, one by judgment.

For a chatbot, the test is the output. For an agent, the test is the path.

05 · Pitfalls

Where agents quietly go wrong.

"Give it everything"

A massive toolset with vague descriptions. Few sharp tools beat many fuzzy ones.

No human in the loop

Day-one agent sends emails to real customers. Stage the autonomy — review, then assist, then act.

Chatbot-shaped evals

"Was the answer good?" is the wrong question. Ask: "was that a defensible sequence of actions?"

No fallback path

When the agent fails, what does the user see? A queue? A human? A nothing? Decide before shipping.

Worked example · 01 of 04

A research agent.

From topic to sourced brief, in steps.

Where the prompting deck synthesized transcripts you already had, this agent goes and gets the material itself — searching, fetching, taking notes, knowing when to stop.

Same goal as a research prompt. Different shape: it's a process, not a single turn.

  • Tools
  • Policy
  • Stop
  • Citations
agent spec GoalProduce a one-page sourced brief on [topic] for an internal PM audience. Toolsweb_search(query) → top 5 results fetch(url) → extracted page text note(claim, source_url) → appends to working brief Policy1. Start with 2 broad searches to map the topic. 2. Then 3 narrow searches for specific claims. 3. Every note() must cite a real source_url returned by fetch(). 4. If two sources disagree, surface the disagreement. Stop - after 8 search-or-fetch calls, OR - when 3 sources confirm the main finding, OR - if 4 searches return nothing useful — return what you have and flag the gap. OutputMarkdown brief with inline citations. Sources at the bottom. Open questions at the end.
Worked example · 02 of 04

An analyst agent.

Estimates with the math actually done.

The prompting deck asked for stated-assumption estimates. The analyst agent goes further: it queries the data, runs the numbers, and writes the report — showing the SQL it used and the formula it ran.

The PM argues with the assumptions, not the conclusion.

  • Tools
  • Code
  • Confidence
  • Table out
agent spec GoalEstimate the impact of the 8 candidate features in this quarter's shortlist, using real numbers from our product data. Toolsquery_db(sql) → read-only, against: product_events, feedback, subscriptions run_python(code) → for calculation & modeling write_report(content) → final markdown out Policy1. For each feature, define a measurable proxy: "% MAU affected" = users who touched [related event] in 30d. 2. Use query_db to count. Never guess a number. 3. Use run_python for revenue projection, with a stated formula. 4. Confidence below 60% → flag ⚠. Stop - all 8 features have an estimate, OR - 3 consecutive queries return empty (data is missing), OR - 20 total tool calls. OutputTable: Feature | %MAU | $/mo | Confidence | Key assumption | Query. SQL of each query appended.
Worked example · 03 of 04

Project files as a constitution.

Standing rules the agent consults every step.

Same files you'd put in a prompt — but the agent reads them before each decision, the way a new hire consults the company handbook before sending anything to a customer.

  • Vision docOne-page product intent
  • OKRsQuarterly outcomes & metrics
  • ICPWho we sell to, who we don't
  • GlossaryTerms, acronyms, naming
  • RoadmapIn-flight and next-up
  • Style guideVoice for customer copy
  • Decision logWhat we chose, why
  • ConstraintsLegal, compliance, tech
agent spec GoalHelp the PM scope any incoming feature request, grounded in the team's standing decisions. Constitutionread before every run - /project/vision.md - /project/okrs-q3.md - /project/icp.md - /project/constraints.md - /project/decision-log.md Toolsread_project(path) draft_brief(content) flag_block(reason) PolicyFor each new request: 1. Match it to the most relevant OKR. If none, say so. 2. Check whether the audience is inside our ICP. If outside, flag. 3. Check constraints.md for any triggers. Be precise. 4. Search decision-log.md for prior versions of this idea. 5. Only after those four checks, draft a feature description in the style-guide voice. StopAfter all five steps complete, or any check returns a hard block. OutputA scoping brief. Every non-trivial claim cites the project file it came from, by filename.
Worked example · 04 of 04

A triage agent.

Many channels in. Sorted signal out. Human in the loop.

Pulls feedback from multiple sources, classifies and clusters, drafts internal summaries and customer replies — but never sends without a person.

The full pattern: tools, bounded autonomy, escalation paths, draft-not-send by default.

  • Tools
  • Cluster
  • Escalate
  • Confirm
agent spec GoalTriage incoming customer feedback from the past 24 hours. Prepare one internal digest + draft replies for human review. Toolsfetch_feedback(source, since) → zendesk, app_reviews, slack classify(text) → bug | feature_request | praise | confused | churn_risk draft_reply(item_id, template) → draft only — never sends escalate(item_id, reason, owner) → opens a ticket, notifies owner Policy1. Pull from all three sources. 2. Classify each item; cluster by theme. 3. churn_risk → escalate to head of CS immediately. 4. bug + severity-high → escalate to on-call. 5. Others → draft a reply in the style-guide voice. 6. Never send. Always present for human confirmation. StopEvery item classified, every escalation opened, every reply drafted. OutputMarkdown digest: counts per class, top 3 themes, draft replies in a table, escalations summary.
In closing

Four things to remember.

01
An agent is a model in a loop with tools — not a chat.
02
Tools are the product surface. Bound them. Review them.
03
Inspect every step. Stage the autonomy. Earn each notch.
04
Eval the trajectory, not just the final answer.

Calm autonomy. Inspectable judgment.
